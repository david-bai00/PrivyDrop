# PrivyDrop — iPad 连接异常调试记录与后续方案（2025-12-02）

> 文档目的：沉淀当前问题背景、定位线索、已实现的调试能力与可执行的后续方案，便于团队成员在不影响线上用户的前提下继续推进与验证。

## 背景与现象

- 近两日反馈：iPad 端加入房间后，双方 UI 均显示“只有您一人在房间内”，未进入“已连接”。
- 发送端控制台在 iPad 离开时有断开日志；`chrome://webrtc-internals` 显示 offer/answer 成功，但 ICE 状态停留在 `new`，存在 `icecandidateerror`。
- 其他平台（安卓、PC、iPhone）基本正常；iPad 在本地 dev 模式（直连同局域网、Google STUN）可成功建立连接。

## 代码与架构入口（速览）

- 前端核心：
  - WebRTC 基类与角色：`frontend/lib/webrtc_base.ts`、`frontend/lib/webrtc_Initiator.ts`、`frontend/lib/webrtc_Recipient.ts`
  - 调度服务与 Store：`frontend/lib/webrtcService.ts`、`frontend/stores/fileTransferStore.ts`
  - 环境与 TURN 配置：`frontend/app/config/environment.ts`
- 后端信令：
  - Socket.IO 路由：`backend/src/socket/handlers.ts`
  - 房间与 Redis：`backend/src/services/room.ts`

## 已实现的调试能力（12-02 已合入测试环境）

- 前端：在不改变业务逻辑的前提下，追加了关键节点上报 `/api/logs_debug` 与控制台打印（无缓冲、无开关）：
  - Socket：connect/disconnect、收到对端 `ice-candidate` 摘要（仅候选类型 typ、协议 proto，避免暴露 IP/SDP）。
  - PeerConnection：`onicecandidate`（本地候选摘要）、`onicecandidateerror`、`ice/signaling/gathering/connection` 状态变更；`handleIceCandidate`/`addQueuedIceCandidates` 的 queued/ok/fail 汇总；DataChannel open/close/error（含 readyState）。
  - 协商流程：join 开始/超时/成功/失败、Initiator/Recipient 侧的 offer/answer 关键点（setLocal/setRemote/emit）。
- 后端：对信令转发追加“单行摘要日志”（join/ready/recipient-ready/initiator-online/offer/answer/ice-candidate/peer-disconnected），便于与前端时序对齐。

> 注：候选仅记录类型与协议，不包含 IP/端口/SDP，满足隐私与合规要求。

## 测试环境结果（12-02）

- iPad 与 PC 在测试环境(与生产环境类似)下能够快速连接；日志中出现 `icecandidateerror Unauthorized/Failed to establish connection`，但仍然通过 host/srflx 候选建立直连。

## 高概率根因假设（按优先级）

1. iPad→TURN(443/TCP/TLS) 在生产路径受阻或证书链问题（测试环境正常但生产异常的网络差异）。
2. 个别时序/信令路径问题（低概率，现有日志可排查）。

## 验证策略（不影响线上用户）

- 使用个人专用房间，避免与其它用户误入（如超长 ID）。
- 仅对目标房间采集调试日志，屏蔽其它房间日志噪音与影响。

### 方案（最终采纳方向）：按房间 ID 精确启用调试日志

- 个人缓存 ID：`155533333ffff`。
- 实施后，只有当房间 ID 等于该值时，前端与后端才打印/上报调试日志；其它房间保持当前行为，不增加日志量，也不改变任何功能。

## 实施计划（待执行）

- 目标

  - 仅对房间 `155533333ffff` 打开调试，生产其余房间零影响。

- 范围/文件

  - 前端：`frontend/lib/webrtc_base.ts`、`frontend/lib/webrtc_Initiator.ts`、`frontend/lib/webrtc_Recipient.ts`、`frontend/lib/webrtcService.ts`
  - 后端：`backend/src/socket/handlers.ts`

- 具体变更

  - 前端
    - 在 `BaseWebRTC` 中新增 `DEBUG_ROOM_ID = '155533333ffff'`、`debugEnabled: boolean` 与 `dbg()`
    - `joinRoom(roomId)` 时：`this.debugEnabled = (roomId === DEBUG_ROOM_ID)`
    - 所有新增日志统一调用 `this.dbg(...)`（内部判断 `debugEnabled` 后再 `postLogToBackend`）；现有上报点替换为 `this.dbg` 以受控。
    - `webrtcService.ts` 新增 `shouldDbg()`，仅当 sender/receiver 任一 `debugEnabled` 时才上报服务侧补充日志。
  - 后端
    - `handlers.ts` 内新增 `const DEBUG_ROOM_ID = '155533333ffff'`；在 `[sig] …` 摘要日志处，只有当事件关联的 `roomId === DEBUG_ROOM_ID` 才 `console.log`；其余房间不打印。

- 验收

  - 使用房间 `155533333ffff` 复现生产问题时，能够完整获得该房间的端到端日志；其他房间不新增日志。
  - 日志能清晰表明失败点：远端候选未到达/未落地、落地失败、ICE 不进入 checking/connected、DTLS/数据通道异常等。

- 回滚
  - 全部是日志受控与条件判断；可随时回滚该改动或将房间 ID 改为不存在值。

## 操作指引（执行与收集）

1. 你个人在生产仅使用房间 `155533333ffff` 进行复现；结束后主动点“离开房间”。
2. 收集以下两类日志并归档：
   - 后端 `[sig] … room:155533333ffff` 摘要；
   - 前端上报 `/api/logs_debug` 的关键行（`[ICE]/[STATE]/[DC]/[Initiator]/[Recipient]` 等）。
3. 根据日志归因：
   - 若候选到达并 `add ok`，但 ICE/Connection 不进入 connected → 检查 TURN 可达性/DTLS。
   - 若持续出现 `Unauthorized` 且无 `relay` 候选落地 → 校验并重建前端，使 `NEXT_PUBLIC_TURN_*` 与 coturn 一致。

## 后续可能的快速验证（仅你个人房间使用）

- 在你同意的前提下，可仅对该房间临时设定 `iceTransportPolicy: 'relay'` 做 AB 验证，进一步确认“必须依赖 TURN 时是否失败”。该改动同样仅对 `155533333ffff` 生效。
